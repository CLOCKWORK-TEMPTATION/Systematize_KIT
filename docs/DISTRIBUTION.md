# التوزيع الرسمي

> هذه وثيقة مرجعية.
>
> للبداية التشغيلية:
>
> ```text
> docs/START_HERE.md
> ```

هذه الوثيقة تضبط المسار الرسمي لبناء حزمة التوزيع الخاصة بالإطار بعد إعادة التموضع.

## الناتج

الأمر الرسمي:

```text
npm run package:dist
```

هذا المسار يمر عبر:

```text
node .Systematize/scripts/node/cli.mjs build-distribution
```

وقبل بناء الحزمة يقوم المحرك نفسه بتشغيل المولدات الرسمية المطلوبة من المصدر.

إذا غيّر هذا التوليد أي ملف مولد تابع للمستودع نفسه، فإن مسار التسليم الرسمي يرفض المتابعة فورًا ولا يبني التوزيع.

الملفات والمجلدات المولدة التي تُعامل كجزء من حالة التسليم الرسمية تُعرَّف صراحة في:

```text
.Systematize/config/distribution-manifest.json
```

ينتج شيئين داخل:

```text
dist/
```

- مجلد توزيع كامل:
  `dist/systematize-framework/`
- حزمة تشغيل:
  ملف
  `.tgz`
  لمحرك
  `Node`

## عقد التثبيت

داخل مجلد التوزيع يوجد مساران رسميان:

```text
install-framework.mjs
install-framework.ps1
```

وكلاهما يشغّل التهيئة الرسمية عبر محرك:

```text
.Systematize/scripts/node/cli.mjs
```

## القاعدة

- التوزيع الرسمي يبني حزمة التشغيل، والوثائق الرسمية، والقوالب، والسكريبتات، وكتالوج حزم التوسعات.
- محتويات التوزيع المعتمدة تُقرأ من:
  `.Systematize/config/distribution-manifest.json`
  ثم تُثبت داخل الحزمة الناتجة عبر:
  `distribution-manifest.json`
- التوزيع الرسمي يرفض الاستمرار إذا جعل التوليد المسبق أي ملف مولد داخل المستودع يخرج من الحالة المسلَّمة قبل بدء التغليف.
- نجاح
  `package:dist`
  لا يُعد صالحًا إلا إذا بقيت الملفات المتتبعة بعده بلا فرق.
- التثبيت الرسمي لا ينسخ منطقًا موازيًا؛ بل يمر دائمًا عبر محرك
  `Node`
  الرسمي.
- التوسعات الافتراضية تُثبت من الكتالوج أثناء
  `init`
  وفق بيانات الحزمة نفسها.

## المنع المحلي قبل الالتزام

ملف المنع المحلي الرسمي المتتبَّع داخل المستودع موجود في:

```text
.Systematize/scripts/hooks/pre-commit
```

ويُثبَّت تلقائيًا داخل مسار hooks الفعلي الخاص بالمستودع عند تشغيل:

```text
npm ci
npm install
```

لأن الحزمة الجذرية تشغّل:

```text
prepare
```

الذي يستدعي:

```text
npm run setup:hooks
```

ويمكن إعادة التثبيت يدويًا عبر نفس الأمر.

هذا الـ
`pre-commit`
يشغّل:

```text
npm run verify
```

وبذلك يمنع إدخال
`commit`
يحمل اختبارات فاشلة أو وثائق مولدة خرجت من المزامنة أو عقود مكسورة قبل أن تصل الحالة إلى
`CI`
أصلًا.

## تسلسل الإثبات النهائي

الحكم التشغيلي النهائي على الجاهزية يثبت فقط إذا نجح التسلسل التالي من نسخة نظيفة:

```text
npm ci
npm run verify
npm run test:powershell
npm run package:dist
npm run verify:clean-tree
npm run verify
```

ويجب أن يبقى خرج:

```text
npm run verify:clean-tree
```

فارغًا بعد مسار التوزيع الرسمي.

ويثبت هذا التسلسل رسميًا داخل
`CI`
على البيئتين:

```text
windows-latest
ubuntu-latest
```

كما تُشغَّل ملفات التثبيت الرسمية المولدة:

```text
install-framework.mjs
install-framework.ps1
```

على هدف نظيف داخل اختبارات التسليم، حتى لا يبقى الحكم معتمدًا على نجاح التغليف وحده.
